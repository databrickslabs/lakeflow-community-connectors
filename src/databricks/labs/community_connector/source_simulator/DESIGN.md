# Source Simulator — Design

## Status

Design phase. **No implementation yet at this path.** The current implementation
lives at `tests/unit/sources/mock_framework/` under the names
`RecordReplayPatch`, `Cassette`, etc. This document describes the target shape
once that code is moved here, renamed to `Simulator`, and extended with active
simulation against a record corpus.

## Purpose

Provide an in-process **source simulator** that stands in for a live source API
in three contexts:

1. **Unit tests** — connector tests run against the simulator with no live
   credentials and no network, producing deterministic results.
2. **End-to-end pipelines** — a Spark Declarative Pipeline running the
   connector's Python data source executes against the simulator. The simulator
   ships with the connector package so it's available on every Spark worker.
3. **Authored test scenarios** — corpora can be hand-written, recorded once
   and committed, or generated by a synthesizer; the simulator serves them with
   real query semantics (filter / sort / paginate).

"Mock" was the original name and it fits the cassette-replay use case. Once the
component serves traffic with query semantics, "mock" is wrong — it's a
simulator.

## Goals

- Connector code runs unchanged in any mode. The simulator integrates by
  patching `requests.sessions.Session.send` — a single choke-point for every
  `requests`-based HTTP call.
- Cassette format and behavior remain backward-compatible with what
  `mock_framework/` ships today. Existing cassettes keep working.
- New "simulate" mode reads a small endpoint spec + a record corpus and serves
  responses with real filter / sort / paginate semantics.
- Spec authoring matches API documentation conventions (one flat list of params,
  one entry per param) so contributors don't have to learn new abstractions.
- E2E pipelines work in-process per worker. No standalone server, no out-of-band
  deployment.

## Non-goals

- Hosting a long-running HTTP listener. The simulator is a process-local
  intercept layer, never a network service.
- 100% API fidelity. We model what connectors actually exercise (filter, sort,
  paginate against record collections) and provide a Python escape hatch for
  endpoints that don't fit the declarative spec.
- Mocking authentication flows. Auth headers are scrubbed at record time and
  ignored at replay/simulate time. OAuth refresh dances are not modeled.
- Covering connectors that don't use the `requests` library. SDK-based
  connectors (e.g. `google-api-python-client`) need a separate adapter; not in
  scope for this design.

## Operating modes

| Mode | Inputs | Behavior | Primary use |
|---|---|---|---|
| `live` | — | No-op patch; calls hit the real API | Validate the connector and the spec against truth |
| `record` | cassette path | Pass through to live; trim + scrub + persist responses | One-time bootstrap to seed a cassette or corpus |
| `replay` | cassette.json | Verbatim playback of recorded responses | Tests pinned to specific recorded data |
| `simulate` | endpoints.yaml + corpus/ | Active filter / sort / paginate against records | Most unit tests + e2e pipelines + varied scenarios |

A **hybrid** form of `simulate` is also supported: simulate where a spec covers
the endpoint, fall back to a cassette for endpoints the spec doesn't model.
This lets a connector adopt the simulator incrementally — model the most-used
endpoints first, let the cassette cover the long tail, eventually drop the
cassette altogether.

   ║                │   ┌──────────┐       ║

The patch is the single integration point. All modes return a real
`requests.Response`, so connectors can't tell anything has been intercepted.

## Endpoint spec

YAML, one file per source, lives next to the corpus. Each endpoint entry
describes:

- **What** — `path` (with `{placeholders}` for path params), `method`, optional
  `corpus` table to draw from.
- **How** — params with semantic roles (filter, sort, pagination).
- **Response shape** — pagination style, default sort, list-vs-single-entity.

Example for two GitHub endpoints:

```yaml
# endpoints.yaml
endpoints:
  - path: "/repos/{owner}/{repo}/commits"
    method: GET
    corpus: commits
    response:
      pagination_style: page_number_with_link_header
      default_sort: "commit.author.date desc"

    params:
      since:    {role: filter,   field: commit.author.date, op: gte}
      until:    {role: filter,   field: commit.author.date, op: lte}
      author:   {role: filter,   field: author.login,        op: eq}
      sort:     {role: sort_by,  options: [author, committer, date]}
      page:     {role: page}
      per_page: {role: per_page, default: 30, max: 100}

  - path: "/user"
    method: GET
    corpus: user
    response:
      single_entity: true   # returns one dict, not an array
```

### Param roles

| Role | Meaning | Required fields | Multiplicity per endpoint |
|---|---|---|---|
| `filter` | Subset corpus by record field | `field`, `op` | Many |
| `sort_by` | Choose the sort field by value | `options` (whitelist) | 0–1 |
| `sort_order` | Choose direction | `options: [asc, desc]` | 0–1 |
| `page` | Page number (1-indexed) | — | 0–1 |
| `per_page` | Page size | `default`, `max` | 0–1 |
| `offset` | Skip N records | — | 0–1 |
| `limit` | Cap response size | `default`, `max` | 0–1 |
| `cursor` | Opaque continuation token | — | 0–1 |
| `ignore` | Accept-but-ignore (e.g. `expand=author`) | — | Many |

Spec authors compose roles to express any common API contract. Unrecognized
roles or fields raise at spec-load time so typos surface immediately.

### Filter operators

`eq`, `ne`, `gte`, `lte`, `gt`, `lt`, `in`, `contains`, `starts_with`,
`ends_with`. Applied to the record's field value with dotted-path traversal
for nested fields (`commit.author.date`). Anything more complex drops to a
custom handler:

```yaml
endpoints:
  - path: "/search/code"
    method: GET
    handler: my_pkg.my_module:search_code_handler
```

A custom handler receives the parsed `Request` and the `CorpusStore` and
returns a `Response`. Reserved for cases the declarative spec can't express
(full-text search, joins, computed fields).

## Corpus

Specs and corpora live **under the simulator package**, organized by source —
not scattered under each connector's source directory. This keeps connector
source trees focused on connector code and gives the simulator package a
single root that owns all simulator artifacts:

```
src/databricks/labs/community_connector/source_simulator/
├── ...framework code...
└── specs/
    ├── github/
    │   ├── endpoints.yaml
    │   └── corpus/
    │       ├── commits.json     # array of commit dicts
    │       ├── issues.json      # array of issue dicts
    │       ├── pulls.json
    │       └── user.json        # single dict (because single_entity: true)
    ├── qualtrics/
    │   ├── endpoints.yaml
    │   └── corpus/
    │       └── ...
    └── ...
```

Because everything is one Python package, the simulator + every source's spec +
corpus all ship together to Spark workers in the e2e use case.

Records are plain JSON: an array of dicts (or a single dict for single-entity
endpoints). Three legitimate ways to populate the corpus:

- **Hand-written** for fixtures and edge cases.
- **Generated** by a synthesizer (deterministic per seed).
- **Bootstrap from one-time recording**: run live once, scrub, save to corpus,
  commit. After this, no live calls are needed.

The corpus is intentionally **decoupled** from cassettes. Cassettes capture
per-request HTTP traffic; corpora capture *data*. The simulator builds
responses from the data + the endpoint spec, so the same corpus serves any
combination of filters and pagination the spec declares.

## Pagination styles

Each style is a plug-in implementing two methods:

```python
class PaginationStyle(Protocol):
    def parse_request(
        self, query: dict, headers: dict, spec: PaginationSpec
    ) -> tuple[int, int]:
        """Translate request → (offset, limit) into the corpus."""

    def render_response(
        self, page: list, total: int, offset: int, limit: int,
        request_url: str, base_url: str,
    ) -> tuple[Any, dict]:
        """Build (response_body, response_headers) for this page."""
```

Built-in styles (covering the connectors in this repo today):

- `page_number` — `?page=N&per_page=M`. Body is the records array.
- `page_number_with_link_header` — same as above, plus a GitHub-style
  `Link: <...>; rel="next", <...>; rel="last"` header.
- `offset_limit` — `?offset=N&limit=M`. Body may be the array or wrapped
  (configurable).
- `cursor` — opaque `?cursor=...`; cursor encodes the next offset; appears as
  `next_cursor` in the response body.
- `odata_next_link` — OData v2/v4: response includes `@odata.nextLink`.
- `none` — single page, no pagination (small endpoints, single entities).

New styles are dropped into the package as one file each; spec authors
reference them by name.

## Public API

```python
from databricks.labs.community_connector.source_simulator import Simulator, Mode

# Replay (unit-test default — same shape as today's RecordReplayPatch)
with Simulator(mode=Mode.REPLAY, cassette_path=p):
    connector.read_table(...)

# Simulate (new — for e2e and varied data)
with Simulator(mode=Mode.SIMULATE, spec_path=s, corpus_dir=c):
    pipeline.run()

# Hybrid: simulate where the spec covers; cassette fallback elsewhere
with Simulator(
    mode=Mode.SIMULATE,
    spec_path=s, corpus_dir=c,
    fallback_cassette=p,
):
    connector.read_table(...)
```

`LakeflowConnectTests` (the test harness) gains optional class attributes:

```python
class TestGithubConnector(LakeflowConnectTests):
    connector_class = GithubLakeflowConnect
    simulator_source = "github"   # resolves to source_simulator/specs/github/
```

The harness resolves `simulator_source` against the simulator package's
`specs/` root, picking up `endpoints.yaml` + `corpus/` for that source.

When `simulator_spec` is set, the harness picks `Mode.SIMULATE`. The two
existing class attributes (`record_replay_*`) keep working unchanged for tests
that don't migrate.

```

├── replayer.py                  # Cassette → response (split from recorder)

├── synthesizer.py               # Variation (existing)
│

├── handler.py                   # Match request → spec → corpus → response

```

Flat layout. Split into subpackages later if any file grows past ~400 lines.

## Design choices and rationale

### Patch `requests.sessions.Session.send`?

Every `requests.get` / `Session.post` / module-level `requests.request` funnels
through `Session.send(PreparedRequest)`. Patching it once catches every
connector that uses the `requests` library. Connectors require zero changes —
the patch is enabled by entering the `Simulator` context manager.

Tradeoff: connectors using a non-`requests` HTTP client (Google's
`google-api-python-client` for Analytics) bypass the patch and need a separate
adapter. We accept this — they're rare, and a per-SDK patch is a small
additive change later.

### Why YAML for the spec, JSON for the corpus?

YAML matches the existing `connector_spec.yaml` convention in this repo and
permits inline comments, useful for documenting per-endpoint quirks. The corpus
is data, not configuration — JSON is the natural format for record arrays,
easy to diff, search, and machine-process.

### Why per-param `role` instead of separate `filters` / `sort` / `pagination` sections?

Earlier draft had three top-level sections. From the API's perspective they're
all just query params; making authors learn three sub-schemas is friction with
no payoff. A single flat `params:` list mirrors how API docs describe
endpoints and makes adding a new param semantic later (e.g. `expand`) a
one-line addition (a new `role:` value).

The simulator still internally runs a filter → sort → paginate pipeline; it
just buckets params by role at spec-load time. That's an implementation
detail, not something authors see.

### Why isolate response shape from params?

Pagination *style* (Link header vs `next_cursor` body field vs OData
`@odata.nextLink`) and `single_entity` are not param-level concerns — they
describe how the endpoint *renders responses*, not how it accepts requests.
Mixing them into the params list would be confusing.

### Why keep cassette mode at all?

Three real uses:

1. **Simple unit tests.** When all you need is "this connector calls the right
   URL with the right params and gets back a reasonable response", a cassette
   is faster to author than a full spec + corpus. Authoring a one-off cassette
   should remain easy and well-documented.
2. **One-time bootstrap.** Record live, scrub, convert to corpus once. The
   cassette is the intermediate format.
3. **Fallback during incremental migration.** Hybrid mode lets a connector
   simulate the most-used endpoints while the cassette covers the rest.

Cassette mode is a peer of `simulate`, not a deprecated path.

### Why is the simulator under `src/`, not `tests/`?

So Spark workers can import it. When a Python data source runs in an e2e
pipeline, the simulator code rides along in the connector package and is
available in every worker. A test-only location would prevent this. Putting
the simulator alongside the connector code (not next to the connector
implementations under `sources/`, but at the namespace root) keeps it usable
from any caller without a separate distribution.

### Why ship via the connector's package?

The connector package is what the worker imports. Putting the simulator in
the same Python namespace (`databricks.labs.community_connector`) means
workers get the simulator for free as a transitive import, with no extra
packaging or deployment.

### Why no synthesis at replay time?

Determinism. Tests that compare exact records (write-back round-trips, "did
this row appear in the read?") need a stable corpus. Synthesis is supported,
but the synthesized output is committed to a file and reused — never
regenerated per test run. This also means the cassette and corpus formats
remain hand-inspectable and diffable.

## Migration plan

| Phase | Scope | Risk | Mergeable independently? |
|---|---|---|---|
| **1. Move + rename** | `tests/unit/sources/mock_framework/` → `src/.../source_simulator/`. `RecordReplayPatch` → `Simulator`. All imports updated. Behavior unchanged. | Low — purely mechanical. | Yes |
| **2. Add corpus mode** | New modules: `endpoint_spec.py`, `corpus.py`, `handler.py`, `pagination.py`. New `Mode.SIMULATE`. Unit and integration tests for each new module. | Medium — new code, but additive (cassette mode untouched). | Yes |
| **3. Convert GitHub** | Author `endpoints.yaml`. Generate synthetic `corpus/*.json` (no PII, deterministic). Switch `TestGithubConnector` to simulate mode. Delete the cassette. | Medium — touches one consumer; proves the design. | Yes |
| **4. E2E example** | Standalone script or notebook running an SDP pipeline against the simulator in-process. Documents the e2e workflow. | Higher — new territory. | Yes |

## Open questions

1. **Filter dotted-path semantics for arrays.** `parents[0].sha` (index) vs
   `parents.sha` (any element)? Lean toward not modeling array indexing in v1
   — escape hatch is a custom handler.
2. **Sort stability.** When two records share the sort field, do we want a
   stable secondary sort (e.g. by primary key)? Most APIs do this implicitly.
   Should we?
3. **Hybrid fallback policy.** When a request matches both a spec endpoint and
   a cassette entry, which wins? Lean toward spec-first (more flexible), with
   an opt-in flag to flip the precedence.

### Resolved

- **Where do per-source specs live?** Under
  `source_simulator/specs/<source>/`, not under each connector's source
  directory. Keeps connector source trees focused on connector code; gives
  the simulator package a single root that owns all spec artifacts; ships
  with the connector package for e2e workers.
- **Long-term role of cassette mode.** Kept as a peer of `simulate`, not a
  deprecated path. Useful for simple unit tests, one-time bootstrap, and
  hybrid migration fallback.
